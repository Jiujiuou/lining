# 多商品监控改造方案

## 一、现状与目标

- **现状**：只监控一个商品（主商品 ID 如 `1017849608938`），三条链接中：
  - **商品加购件数**：`/cc/item_rank?dateRange=...&dateType=today`（当前无 itemId，按页面上下文理解为「当前页商品」）
  - **某商品其余字段（流量来源等）**：`/cc/item_archives?activeKey=flow&dateRange=...&dateType=today&itemId=1017849608938`
  - **店铺排名**：不变，暂不修改
- **目标**：支持监控**多个商品**，每个商品有固定主商品 ID；从「插件补货 → 存储 → 数据表展示」全链路支持多商品。

---

## 二、整体改造思路

1. **商品列表**：在扩展内维护「监控商品列表」（itemId + 可选昵称），用于生成要打开的链接和前端展示名称。
2. **补货链接**：按商品列表，为每个商品生成「加购页 + 流量来源页」两个链接；店铺排名仍只生成一条。
3. **抓数时带上商品**：在打开某商品页面时，从 URL 解析出 `itemId`，随事件一起传给 content，写入库时带上 `item_id`。
4. **节流按商品区分**：同一时段内「同一商品」只写一次，不同商品独立节流。
5. **数据库**：在加购表、流量来源表增加 `item_id` 字段。
6. **前端**：按 `item_id` 拉数、分组，用「当前选中商品」或多 Tab/多区块展示。

---

## 三、分模块改造

### 3.1 插件 Popup（补货 + 商品配置）

| 项 | 说明 |
|----|------|
| **商品列表存储** | 使用 `chrome.storage.local` 存列表，例如 key：`sycm_monitor_items`，值：`[{ itemId: '1017849608938', name: '小贝壳' }, ...]`。默认可保留一个商品，兼容旧行为。 |
| **Popup UI** | 增加「监控商品」区块：展示列表、支持「添加」（输入 itemId + 可选昵称）、删除。与「打开生意参谋页面」按钮同屏。 |
| **getSycmLinks()** | 1）从 storage 读取商品列表；2）`dateRange` 仍用东八区今天；3）对**每个商品**生成两条链接：<br>• 加购：`https://sycm.taobao.com/cc/item_rank?dateRange=...&dateType=today&itemId=XXX`（若生意参谋该页支持 itemId；若不支持，需确认是否只能通过「进入某商品详情」再拿到该商品的 top.json，届时改为「每个商品一个 item_archives 链接」触发上下文）；<br>• 流量来源：`https://sycm.taobao.com/cc/item_archives?activeKey=flow&dateRange=...&dateType=today&itemId=XXX`；<br>4）最后追加**一条**店铺排名链接（不变）。 |
| **打开逻辑** | 「打开生意参谋页面」仍为一次打开多个 Tab；链接顺序建议：先 1 个店铺排名，再按商品顺序每个商品 2 个 Tab（加购 + 流量来源），避免同时开太多可考虑间隔或分批。 |

**注意**：若实际验证发现 `item_rank` 的 URL 不支持 `itemId`，则加购数据只能通过「打开带 itemId 的页面（如 item_archives）后，该页是否也会请求 top.json」来获取；若只有 item_archives 页会按 itemId 请求，则每个商品只打开一条 item_archives 链接，加购与 flow 可能来自同一页的不同接口（需再确认 top.json 是否在 item_archives 页也会被请求）。

---

### 3.2 Inject 与 Content（抓数 + 写库带 item_id）

| 位置 | 改造点 |
|------|--------|
| **inject.js** | 在 `handleResponse` 里，当命中「商品相关」的 URL（如包含 `item_archives` 或 `item_rank`）时：从 `window.location.search` 解析 `itemId`（如 `itemId=1017849608938`）。派发 CustomEvent 时在 `detail` 中增加 `itemId`（字符串）。若当前页 URL 没有 itemId，可传 `itemId: null`，由 content 写库时用默认或忽略。 |
| **content.js** | 1）`handleEvent` 里读取 `d.itemId`；2）写 Supabase 的单条/批量记录时，在 `record` 中增加 `item_id: d.itemId || ''`（或业务约定的默认值）；3）节流槽 key 从 `lastSlotPrefix + eventName` 改为 `lastSlotPrefix + eventName + '_' + (d.itemId || '')`，实现「按商品+事件+时段」去重。 |
| **config.js** | pipelines 中 cart、flow 的 `extractValue` 无需改；仅 inject 在派发时补上 itemId 即可。店铺排名 pipeline 不改。 |

---

### 3.3 数据库（Supabase）

| 表 | 变更 |
|----|------|
| **sycm_cart_log** | 新增列：`item_id text`。可为 `null` 或设默认值以兼容历史数据（如 `default '1017849608938'`）。 |
| **sycm_flow_source_log** | 同上：`item_id text`。 |
| **sycm_market_rank_log** | 不改。 |
| **RLS / Realtime** | 无需改，仍允许 anon 插入与查询。 |

迁移示例（在 Supabase SQL Editor 执行）：

```sql
-- 加购表
alter table public.sycm_cart_log
  add column if not exists item_id text default '1017849608938';

-- 流量来源表
alter table public.sycm_flow_source_log
  add column if not exists item_id text default '1017849608938';
```

### 3.4 前端数据表展示（React App）

| 项 | 说明 |
|----|------|
| **拉数** | `loadFromSupabase` 中 `supabase.from('sycm_cart_log')` 与 `sycm_flow_source_log` 的 `select` 增加 `item_id`；条件仍按 `created_at` 日期区间。 |
| **状态结构** | 可保留「按商品聚合后的数据」：例如 `rawCartLogRows` / `rawFlowSourceRows` 仍为数组，每行带 `item_id`；在 useMemo 中按 `item_id` 分组得到 `parsedDataByItem: { [itemId]: { dates, byDate } }`。 |
| **商品列表来源** | 方案 A：前端从 Supabase 的「商品主表」读（需新建表 `sycm_items(id, item_id, name)` 并由 popup 或后台维护）。方案 B：前端从 localStorage / 环境变量读 itemId→昵称映射。方案 C（最小实现）：不做昵称，仅按 `item_id` 展示，图表标题用「商品 {item_id}」或「商品 1/2/3」。 |
| **展示方式** | 1）**商品选择器**：顶部增加下拉或 Tab「当前商品」，选中后下方图表只展示该商品的加购 + 流量来源；2）或**多区块**：一页内每个商品一个折叠/区块，各自一套图表。店铺排名仍全局一份，不按商品切。 |
| **图表 key** | 当前写死「小贝壳-商品加购件数」等，改为按商品区分：如 `{商品昵称或 item_id}-商品加购件数`，避免多商品时 key 冲突；`mergeCartAndFlowChartData` 等按单商品合并即可，多商品时在更上层按 item_id 分组后再合并。 |
| **Realtime** | 订阅 INSERT 时，新行带 `item_id`，前端按 `item_id` 归入对应商品的 `rawCartLogRows` / `rawFlowSourceRows` 再触发重算。 |

---

## 四、实施顺序建议

1. **数据库**：先加 `item_id` 列并做默认值迁移。
2. **扩展**：  
   - 先做 storage 商品列表 + popup UI + `getSycmLinks()` 按商品生成链接；  
   - 再改 inject 解析 URL 的 itemId 并写入 detail；  
   - 再改 content 写库带 `item_id`、节流按 `eventName_itemId`。
3. **前端**：  
   - 先拉数带 `item_id`、按 `item_id` 分组得到多份 parsedData；  
   - 再加「当前商品」选择与单商品图表展示；  
   - 最后可选：商品主表 + 昵称、多区块/多 Tab 等。

---

## 五、小结

| 环节 | 改造要点 |
|------|----------|
| **插件补货** | 商品列表存 chrome.storage；getSycmLinks 按列表为每个商品生成「加购 + 流量来源」链接，并保留一条店铺排名。 |
| **抓数与写库** | inject 从当前页 URL 取 itemId 放入事件 detail；content 写 Supabase 时带 `item_id`，节流 key 含 itemId。 |
| **数据库** | sycm_cart_log、sycm_flow_source_log 增加 `item_id`；店铺排名表不动。 |
| **数据表展示** | 查询带 `item_id`，按 item_id 分组；增加商品选择或多区块，图表 key 按商品区分。 |

店铺排名逻辑与表结构均不修改，仅对「商品加购件数」与「流量来源」两条链路做多商品扩展即可。
