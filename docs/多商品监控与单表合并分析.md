# 多商品监控与单表合并分析

## 一、数据源与现状

| 来源 | 接口 | 数据内容 | 当前落库 |
|------|------|----------|----------|
| **A - 加购** | 多商品加购接口（一次返回所有监控商品的加购数） | 每商品：`itemId` + `itemCartCnt`（加购件数） | `sycm_cart_log`（单表，无 item_id） |
| **B - 详情** | 每商品单独详情接口（需打开该商品详情页） | 该商品：搜索访客数、搜索支付转化率、购物车访客数、购物车支付转化率 | `sycm_flow_source_log`（单表，无 item_id） |

- 补货方式：**1 个页面**（多商品加购） + **N 个页面**（N 个商品各一个详情页）。
- 两路数据都按 **20 分钟** 时间间隔节流写入。

多商品加购接口返回结构（与 `response.json` 一致）：

- 路径：`data.data.data` 为数组。
- 每项含：`item.itemId` 或 `itemId.value`（商品 ID）、`itemCartCnt.value`（加购件数），字段含义与现有单商品一致。

## 二、从单商品到多商品：方案是否可行？

**结论：可行。**

1. **多商品加购接口**
   - 在 config 中新增一条 pipeline，`urlContains` 匹配该接口的 URL（需在生意参谋里从网络请求确认具体路径）。
   - `extractValue`：遍历 `data.data.data`，对每项取 `item_id`（`item.itemId` 或 `itemId.value`）和 `item_cart_cnt`（`itemCartCnt.value`），返回多行或 `multiRows: true` 的 payload。
   - content 侧：按「每行带 `item_id`」写入；节流 key 使用 `eventName + '_' + item_id`，保证同一 20 分钟槽内同一商品只写一次。

2. **详情数据（搜索访客数、转化率等）**
   - 保持现有「流量来源」pipeline，从当前页面 URL 解析 `itemId`，写入时带上 `item_id`。
   - 打开 N 个详情页，就得到 N 个商品的详情数据；节流同样按 `(eventName, item_id, slotKey)` 区分。

3. **补货链接**
   - `getSycmLinks()`：生成 **1 条**多商品加购页 URL + **N 条**详情页 URL（每商品一条带 `itemId` 的详情页）。
   - 用户点击「打开生意参谋页面」后：1 + N 个 Tab，对应 1 次多商品加购请求 + N 次详情请求。

只要在多商品加购页的请求里确认到具体 URL（如某 rank/list 接口），即可在 inject 里匹配并解析，无需改业务逻辑本质。

## 三、合并两张表是否合适？

### 3.1 你的判断

> 这些内容都是「某个商品在某个时间段」的数据，不需要分到两张表；合并成一张表后，数据表渲染时也方便统一来源。

从语义和前端使用角度，这个判断是成立的。

- 两表当前维度一致：**商品 + 时间**（且时间都按 20 分钟对齐）。
- 前端展示时本来就要按「商品 + 日期 + 时段」把加购和流量来源拼在一起；若仍用两表，就要按 `item_id` + 时间槽做 join 或内存对齐，逻辑重复、易错。
- 合并成一张「宽表」后：一行 = 一个商品在一个 20 分钟时段的所有指标，查询一次、按 `item_id` 分组即可渲染，数据来源统一。

### 3.2 合并后的唯一难点：双源写同一张表

- **来源 A（加购）**：只提供 `item_id`、时间槽、`item_cart_cnt`。
- **来源 B（详情）**：只提供 `item_id`、时间槽、`search_uv`、`search_pay_rate`、`cart_uv`、`cart_pay_rate`。

同一行（同一 `item_id` + 同一 20 分钟槽）可能先被 A 写入、再被 B 更新，或反过来。因此不能只用「简单 INSERT」，而要用 **UPSERT**：

- 主键（或唯一约束）：`(item_id, slot_ts)`，其中 `slot_ts` 为 20 分钟对齐的时间（如 `timestamptz` 或统一格式的字符串）。
- **A 写入**：有则更新 `item_cart_cnt`，无则插入一行（其它指标可为 null）。
- **B 写入**：有则更新 4 个流量指标，无则插入一行（加购可为 null）。

这样「按 20 分钟间隔」的两路数据会自然合并到同一行，前端只读一张表即可。

### 3.3 合并单表的利弊小结

| 维度 | 说明 |
|------|------|
| **优点** | 前端单一数据源、一次查询、按 item_id 分组即可；模型与「某商品某时段」一致；后续加更多指标仍可落在同一行，扩展简单。 |
| **代价** | 写入从「简单 INSERT」改为「按 (item_id, slot_ts) UPSERT，且只更新本来源负责的列」；扩展与 Supabase 需支持 upsert（Supabase 支持 `insert + onConflict`）。 |
| **结论** | **合并成一张表是更优方案**，只要约定好 `slot_ts` 的取整规则（两源一致），并实现按列部分更新的 upsert 即可。 |

## 四、单表设计建议（宽表 + 20 分钟槽）

### 4.1 表结构

建议新表名例如：`sycm_item_slot_log`（与现有两表区分，便于迁移或并行一段时间）。

```sql
create table public.sycm_item_slot_log (
  id bigint generated always as identity primary key,
  item_id text not null,
  slot_ts timestamptz not null,   -- 20 分钟对齐的时段开始时间，如 2026-02-26 09:00:00+08
  item_cart_cnt integer null,     -- 来源 A
  search_uv integer null,
  search_pay_rate numeric null,
  cart_uv integer null,
  cart_pay_rate numeric null,    -- 以上 4 列来源 B
  created_at timestamptz default now(),
  updated_at timestamptz default now(),
  unique (item_id, slot_ts)
);

-- RLS、policy、realtime 同现有表思路
alter table public.sycm_item_slot_log enable row level security;
create policy "Allow anon insert for sycm_item_slot_log" on public.sycm_item_slot_log for insert to anon with check (true);
create policy "Allow anon select for sycm_item_slot_log" on public.sycm_item_slot_log for select to anon using (true);
create policy "Allow anon update for sycm_item_slot_log" on public.sycm_item_slot_log for update to anon using (true) with check (true);
alter publication supabase_realtime add table public.sycm_item_slot_log;
```

- `slot_ts`：由 `recordedAt` 按 20 分钟向下取整得到，**A、B 两源使用同一套取整规则**（例如东八区 9:00 起算，9:00、9:20、9:40…）。

### 4.2 写入语义（扩展侧）

1. **加购源 A**  
   - 对多商品接口返回的每条 `(item_id, item_cart_cnt)`，计算同一 `slot_ts`。  
   - 执行 upsert：`ON CONFLICT (item_id, slot_ts) DO UPDATE SET item_cart_cnt = EXCLUDED.item_cart_cnt [, updated_at = now()]`，不覆盖 B 的列。

2. **详情源 B**  
   - 对当前详情页的 `(item_id, search_uv, search_pay_rate, cart_uv, cart_pay_rate)`，计算同一规则的 `slot_ts`。  
   - 执行 upsert：`ON CONFLICT (item_id, slot_ts) DO UPDATE SET search_uv = EXCLUDED.search_uv, ...`，不覆盖 `item_cart_cnt`。

Supabase 的 upsert：  
- 使用 `insert().upsert({ ... }, { onConflict: 'item_id,slot_ts' })`，并只传入本来源有的列，其它列在 conflict 时用 `ignoreDuplicates: false` + 仅更新传入列即可（或写 raw SQL 的 `DO UPDATE SET` 只更新非空列）。

### 4.3 前端

- 只查 `sycm_item_slot_log`，按 `item_id` 分组。
- 每个商品、每个日期的 20 分钟槽，一行里同时有加购与 4 个流量指标；现有「加购 + 流量」两张图的渲染逻辑可统一成：从同一批 rows 按 `item_id` 和 `slot_ts` 聚合并拆成「加购」与「流量」系列，无需再 join 两表。

## 五、总结

| 问题 | 结论 |
|------|------|
| 从监控单商品到多商品是否可行？ | **可行**：加购用多商品接口批量拿并带 `item_id` 写入；详情保持每商品一页、带 `item_id` 写入；补货 1 + N 个页面即可。 |
| 合并两张表是否是好方案？ | **是**：语义上是「某商品某时段」一条记录，合并后前端单一来源、渲染简单；代价是写入改为按 `(item_id, slot_ts)` 的 upsert，两源各写自己负责的列。 |
| 两源、20 分钟间隔能否做到？ | **可以**：用统一 `slot_ts`（20 分钟对齐）+ 唯一约束 `(item_id, slot_ts)` + upsert，A 只更新加购列，B 只更新流量列，即可在单表内按时间段合并。 |

下一步需要你在生意参谋里确认「多商品加购」接口的完整 URL（例如是否包含 `/cc/item_rank` 或某个 list 接口路径），以便在 config 的 `urlContains` 里配置；拿到 URL 后即可在扩展里加新 pipeline、并实现上述单表 upsert 与前端单表查询。
